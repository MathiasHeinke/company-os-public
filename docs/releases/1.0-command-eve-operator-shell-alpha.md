# Company.OS 1.0 Command EVE Operator Shell Alpha

Status: alpha notes for `1.0.0-alpha.3`
Audience: a remote founder/operator installing Company.OS plus local Command
EVE AionUI/Hermes sidecars.
Last updated: 2026-06-05

## What this is

`1.0.0-alpha.3` is the current local-first Command EVE operator-shell alpha. It
keeps the Alpha.2 self-install path and adds the 0.8/0.9 closure layer:
department dashboard/review-card, support/security/privacy, scheduler
kill-switch, context-topology, watchdog, connector-harness and remote-pilot
handoff work are now explicit public child contracts with CAO security/
code-review/hotfix gates.

The public clone remains the source. After the Company.OS workspace is
initialized, the operator can install:

- AionUI `v2.1.10` from source with the Command EVE overlay applied.
- AionCore `v0.1.19` prepared by AionUI's own pinned backend downloader and
  wired into the generated launcher.
- AionUI renderer assets built locally so the generated `start_eve` launcher
  can run the web UI without a separate manual build.
- Hermes Agent `0.15.2` in a local Python venv.
- EVE SOUL, runtime policy, connector manifest and first-run skill pack.
- A full German-first Command EVE AionUI language pack: `de-DE` covers every
  AionUI `en-US` namespace with key/placeholder parity, is the fresh-install
  default, and keeps `en-US` selectable for operators who prefer English.
- AionUI login branding aligned to `try.command-eve.com`, with a muted
  autoplaying face-focused EVE wait loop, `⌘ EVE` product identity and a
  separate generated `<installed-version>` line from
  `public/command-eve-brand.json`.
- Default inference profile: `openrouter` / `minimax/minimax-m3`, with a
  BYOK/auth preflight receipt that reports `BLOCKED_AUTH` for provider-key or
  quota problems instead of treating the product install as broken.
- First-run EVE confirmation receipt: EVE can state known account/company facts
  from the installed seed, ask for correction, then guide the next setup step
  without opening with a broad questionnaire.
- Client-local launchers: `.company-os/bin/start_eve` and
  `.company-os/bin/update_eve`.

This is still alpha. It is designed for a supervised remote founder call, not
an unsupported stable desktop product.

## Commands

```bash
git clone https://github.com/MathiasHeinke/company-os-public.git
cd company-os-public

node scripts/install/command-eve-self-install.mjs install --dry-run \
  --target /path/to/company-workspace \
  --company "Acme Systems" \
  --website "https://acme.example" \
  --offer "AI operating-system setup" \
  --buyer "founder-led service firms" \
  --founder "Jane Founder" \
  --approval-owner "Jane Founder" \
  --first-department marketing \
  --json

node scripts/install/command-eve-self-install.mjs install \
  --target /path/to/company-workspace \
  --company "Acme Systems" \
  --website "https://acme.example" \
  --offer "AI operating-system setup" \
  --buyer "founder-led service firms" \
  --founder "Jane Founder" \
  --approval-owner "Jane Founder" \
  --first-department marketing \
  --json
```

The first command is a no-write dry-run. It checks signup seed, source version,
target writability, existing install state, `git`, `python3`, `bun`, pinned
AionUI/Hermes component versions and generated start/update command targets
before any sidecar state is written.

The second command runs the public-RC workspace install and then the managed
Command EVE operator-shell install. If it passes, it prints the local AionUI
start command. The operator may then open AionUI and start EVE with the
first-run boot packet.

For manual debugging during a remote support call, the wrapper can be split
into the two lower-level stages:

```bash
node scripts/install/public-rc.mjs install \
  --target /path/to/company-workspace \
  --company "Acme Systems" \
  --website "https://acme.example" \
  --offer "AI operating-system setup" \
  --buyer "founder-led service firms" \
  --founder "Jane Founder" \
  --approval-owner "Jane Founder" \
  --first-department marketing \
  --json

node scripts/operator-shell/install-command-eve.mjs install \
  --client-root /path/to/company-workspace \
  --write-report \
  --json
```

The localization gate can be run after the managed AionUI sidecar exists:

```bash
node scripts/operator-shell/aionui-localization-gate.mjs \
  --aionui-root /path/to/company-workspace/.company-os/operator-shell/aionui-sidecar/AionUi \
  --json
```

It fails if upstream AionUI adds/removes locale keys or if the Command EVE
`de-DE` pack loses a required `{{placeholder}}`.

The install also writes shorter local commands into the target workspace:

```bash
/path/to/company-workspace/.company-os/bin/start_eve
/path/to/company-workspace/.company-os/bin/update_eve check
/path/to/company-workspace/.company-os/bin/update_eve apply
```

`start_eve` starts the patched AionUI web UI with Hermes as the default EVE
agent. Before AionUI preflight, it refreshes
`public/command-eve-brand.json` from the root `VERSION`, so the login UI cannot
keep showing a stale product version after a repo update. `update_eve check`
pulls the public source when it is a Git checkout and then reports available kit
changes; exported mirrors skip only the pull step. `update_eve apply` applies
safe kit updates and refreshes the pinned AionUI/Hermes sidecars after operator
approval.

Before the first real chat, run the auth/model preflight once:

```bash
/path/to/company-workspace/.company-os/bin/start_eve --auth-check
```

or from the source checkout:

```bash
node scripts/operator-shell/start_eve.mjs check \
  --client-root /path/to/company-workspace \
  --auth-check \
  --json
```

The JSON reports provider, model, auth mode and a short readiness proof. A
provider-key, quota or timeout problem is reported as `BLOCKED_AUTH`; the next
step is to configure OpenRouter or the selected provider through its official
BYOK/API-key flow. Do not paste raw API keys into EVE, Codex, Plane, reports or
the repository.

## Dependencies

The `source-overlay` lane clones AionUI, installs dependencies, prepares the
pinned AionCore backend binary and runs the AionUI renderer build. On a fresh
machine this can take a few minutes. It
requires:

- `git`
- `python3`
- `bun`

The installer reports missing prerequisites before writing sidecar state. On a
fresh macOS machine, `bun` is usually the non-default dependency:

```bash
brew install oven-sh/bun/bun
# Alternative: install Bun from https://bun.sh/docs/installation after verifying
# the official installer for your platform.
export PATH="$HOME/.bun/bin:$PATH"
bun --version
```

The default model is the OpenRouter route `minimax/minimax-m3`; the alpha does
not collect or store raw API keys in Company.OS. The operator authenticates via
the provider's official BYOK flow.

`start_eve --auth-check` is the supported local proof. It emits the selected
provider/model and readiness status, but never prints or requests provider
secrets.

## Boundaries

- No raw API keys in chat.
- No Plane Done.
- No worker dispatch before CEO/Codex review.
- No publish/send/spend/deploy.
- AionUI/Hermes native full-auto, team mode, cron and broad delegation remain
  disabled or HumanGate-gated.

## Why alpha

This release does not yet provide a signed macOS app, hosted tenant,
one-click OAuth provisioning, notarized DMG, bundled account creation or stable
unsupported self-serve claims. It provides a reproducible local install path
that can be tested with the first external founder/operator.

## Alpha.2 Delta

`1.0.0-alpha.2` is the first line where self-install, BYOK auth preflight and
first-run EVE confirmation are all visible in the public install path:

- `VERSION` and `registries/operator-shell/command-eve-1.0-alpha.json` identify
  the public line as `1.0.0-alpha.2`.
- `preflightEveSidecar` and `start_eve --auth-check` emit a Hermes BYOK auth
  profile with provider, model, auth mode, secret policy and `BLOCKED_AUTH`
  next actions.
- The generated runtime boot packet includes
  `first_run_confirmation.known_facts`, `missing_or_unverified_facts`,
  `progressive_setup_queue` and `existing_system_inventory`.
- EVE's first-run contract remains proposal-only: it may draft Founder Intent
  Packet, CEO Delegation Packet and `dispatch: manual` worker contracts, but it
  cannot dispatch workers, mark Plane Done or mutate connectors.
