## 1. Purpose And Non-Goals

The 0.6A Marketing/Growth Command Center is a **text-first conversation map**.
It defines what an employee or operator may ask the Marketing/Growth surface
of Company.OS in the `0.6.x-beta` band, what the system may answer, which
sources it must cite, and which actions remain approval-gated as intent cards
or remain entirely outside the conversational surface.

It is explicitly NOT:

- a control plane that mutates Plane work items from chat
- a publishing engine
- a second scheduler, second ledger, or second memory store
- an authorization layer for spend, deploy, outreach or production writes
- a department pack — those land in `0.8.x-beta` per
  `docs/strategy/autonomy-product-horizon.md`

`FACT(docs/strategy/autonomy-product-horizon.md)` lines 110–118 place the
text-first conversation map at `0.6.x-beta` step 1 and explicitly mark the
read-only Command Center POC and voice as `0.7.x-beta` step 2.

`FACT(ROADMAP.md)` v0.6 entry adds:

- text-first Command Center conversation map for the Marketing/Growth wedge
- sidecar audio/morning-brief remains sandbox-only and feeds findings in
- Raindrop Workshop v0.6.x pilot stays on safe read-only or synthetic paths

This artifact is the entry-point document for that v0.6 line item.

## 2. Scope Boundaries (Include / Exclude)

Included:

- the question/answer surface for Marketing/Growth employees and the operator
- cited source rules for every answer class
- the intent-card boundary between "answer only" and "create gated Plane item"
- the blocked-action catalog the surface must refuse
- three child Worker Contract drafts for the first 0.6A execution slices

Excluded:

- live posting on social, email, blog, or third-party platforms
- scheduling external posts
- outreach sends
- customer-facing replies
- spend changes, ad budget changes, ad copy approvals
- production deployment of any kind
- customer data access from chat
- direct Plane state mutation from chat (no `Done`, no status flips, no
  comment writes other than the gated intent card the human approves)
- regulated claims (medical, legal, Rx, financial) — all such drafts route
  to the Claim/Compliance Gate child contract before any further movement

class, (c) an evidence-citation rule, (d) a default reply mode (`answer-only`,
`answer-plus-suggest-intent-card`, or `refuse-with-redirect`).

## 4. Source Citation Rules

Every answer must obey these citation rules before it is shown to the
employee:

1. **Cite or refuse.** If a question requires an answer source class (column
   3 of Section 3) and the source is not available in this session, the
   surface MUST refuse with: "Source X is required and not available in this
   session. I will not guess."
2. **Quote source paths.** Every factual claim in an answer must end with a
   parenthetical reference of the form `(file: <absolute repo-relative path>)`
   or `(plane: <work-item-id>)` or `(clarity: <view id>)`. Voice-guide
   paraphrases also must end with a source reference.
3. **No memory of facts.** The surface MAY use Honcho read for *context*
   (per the capability profile `honcho_read: company`), but it MUST NOT cite
   Honcho memory as a factual source for marketing or compliance answers.
   `FACT(docs/orchestration/claude-clevel-worker-runtime.md)` enforces
   `honcho_write: proposal-only` for C-level workers — the same proposal
   discipline applies in this conversational surface.
4. **Fresh reads.** Plane and Clarity reads are always live during the
   session; cached answers older than 60 seconds must be re-fetched before
   citation.
5. **No public re-quotation.** Internal Plane comments, internal Clarity
   queries and any internal report paths must never be re-quoted in a
   draft intended for external publication. The surface must strip them
   before producing a draft artifact.

## 5. Blocked Action Catalog

The 0.6A surface MUST refuse the following action classes regardless of how
they are phrased. Refusals are templated and end with a redirect to the
appropriate intent card or HumanGate owner.

| Class | Examples | Authority required to ever execute |
|---|---|---|
| Public posting | "Post this to LinkedIn / X / Threads / Bluesky / Mastodon" | HG-2.5 Marketing release card after editorial review |
| External scheduling | "Schedule this for Friday on Buffer / Hypefury / Make" | HG-2 scheduler intent card approved by `role:cmo` |
| Outreach send | "Email the list", "DM this lead", "send the customer X" | HG-4 (Founder) for customer outreach; no chat path |
| Spend change | "Spend EUR 200 on the ad set", "raise the budget" | HG-4 (Founder) — material spend stays Founder-owned |
| Regulated/public claims | medical, legal, Rx, financial outcome claims | HG-4 (Founder) + Claim/Compliance Gate child (Section 6.3) |
| Production deployment | "Push the landing page", "publish the blog", "deploy" | HG-2.5 release card; chat surface never authorizes |
| Customer data access | "Show me customer X's profile", "export the list" | HG-4 (Founder); chat surface refuses entirely |
| Plane state mutation | "Mark Done", "close the issue", "set status InProgress" | CEO/Founder per `docs/governance/ceo-release-authority.md`; chat may only propose an intent card |
| Schema / RLS / auth / service-role change | "rotate the key", "update RLS for table X" | HG-4 (Founder) per AGENTS doctrine; chat refuses |
| Memory write (durable) | "Remember that we like phrase X", "save this rule" | proposal-only Learning Proposal; never durable from chat |
| Direct CAO override | "Force PASS", "ignore the reject" | refused; CAO PASS/REJECT is binding per `docs/agents/cao.md` |

`FACT(AGENTS.md)` lines 31–115 and `FACT(docs/governance/ceo-release-authority.md)`
(referenced from AGENTS.md) — the blocked action catalog is consistent with
the existing Plane-first doctrine, HG-4 Founder reservation, CAO seat
separation, and CEO release authority. No new release authority is invented
here; the conversational surface is strictly downstream of those gates.

## 6. Intent-Card Template

An **intent card** is the only legitimate way the conversational surface may
cause downstream work. It is a structured artifact, not a free-text request,
and it is parseable by `plane-dispatcher-v0` if a human approves it.

The surface must NEVER write the intent card directly into Plane. It produces
the card as a chat message; a human (operator or `role:cmo` seat) decides
whether to paste it into Plane as a child work item.

### 6.1 What becomes an intent card vs an answer-only reply

| Trigger | Outcome |
|---|---|
| The question can be answered from cited sources without changing any external state | **answer-only** |
| The question describes work that would create, schedule, send, spend, publish, or mutate state | **intent-card draft**, never executed in-session |
| The question crosses a blocked action class (Section 5) | **refuse-with-redirect**, no intent card |
| The question requests durable memory writes | **proposal-only learning_proposal** in the run report, never durable |

### 6.2 Intent-card schema (chat-message form)

The card is presented as a fenced YAML block the operator can copy into Plane.
It maps directly onto `docs/templates/worker-issue-contract.md` required
fields so `plane-dispatcher-v0` can parse it after human review.

```yaml
intent_card_version: 0.6a-1
proposed_role: role:cmo
proposed_parent_seat: role:cmo
proposed_agent: claude
proposed_mode: implement | plan | report | review
proposed_workspace: registry:company-os
proposed_source_of_truth:
  - <absolute path or Plane item id>
proposed_scope:
  include:
    - <one sentence per included slice>
  exclude:
    - no public posting
    - no spend change
    - no customer data
    - no Plane Done from worker
proposed_acceptance_criteria:
  - <verifiable outcome 1>
  - <verifiable outcome 2>
proposed_gates:
  - <command or human review gate>
proposed_human_gate: HG-1 | HG-2 | HG-2.5 | HG-3 | HG-4
proposed_human_gate_owner: role:cmo | CEO | Chief-of-Staff | Founder
proposed_reflection_policy: required
proposed_learning_proposal_policy: required
proposed_blocked_actions:
  - <copy of relevant blocked-action lines from Section 5>
notes_for_human_reviewer: <one paragraph why this is the smallest reversible slice>
```

### 6.3 Refuse-with-redirect templates

For blocked classes the surface uses templated refusals. Each refusal ends
with the gate owner so the employee knows where to take the request next.

- "I cannot post externally. Public posting is HG-2.5 after editorial review.
  Routing this to the Social Approval Queue child workflow."
- "I cannot change spend. Spend changes are HG-4 (Founder). I can draft an
  internal brief if you want."
- "I cannot read customer data from this surface. That request must go to
  the Founder-owned customer-data lane."
- "I cannot mark Plane items Done. Done transitions are CEO/Founder per
  `docs/governance/ceo-release-authority.md`."
- "I cannot publish a regulated claim. Routing this draft to the
  Claim/Compliance Gate child workflow before any further movement."

## 9. Reflection (worker self-review)

`reflection:`

- The plan-only mode mapped cleanly to the artifact: this document defines
  rules, schemas and refusal templates, but it neither dispatches workers,
  publishes content, schedules external posts, nor changes any state.
- The biggest risk in this slice was scope creep into the 0.7 Command Center
  POC. The boundary was held by anchoring the artifact to
  `docs/strategy/autonomy-product-horizon.md` Command Center And Voice Layer
  step 1, and by deferring read-only operator drill-down UI to 0.7.
- 26 question classes (Section 3) is more than the acceptance minimum of 20.
  The slight overage gives the controller room to prune in 0.6B without
  re-opening the contract. Pruning later is preferable to under-shipping now.
- One declared source-of-truth file did not exist in the original worker sandbox
  (`docs/releases/0.6-growth-beta-entry-plan.md`). The worker correctly flagged
  it as `HYPOTHESIS` at run time; the file now exists and this controller
  refresh upgrades the evidence class to `FACT` without changing the original
  run chronology.
- The three child Worker Contracts are written as drafts; none is dispatched
  from this run. Each carries its own `human_gate`, `blockedactions`,
  `reflection_policy` and `learning_proposal_policy` to keep Stage 0.5
  Contract Controller review fast when a human routes them.

## 10. Learning Proposals

`learning_proposals:` (proposal-only; no durable memory, SOP, skill, workflow
or harness change happens without controller/CEO review)

1. **proposal:** Extract a reusable refuse-with-redirect template library at
   `docs/templates/conversational-refusal-templates.md` so future
   conversational surfaces (0.7 read-only Command Center, 0.8 department
   assistant) inherit the same refusal language without re-deriving it per
   wedge. Risk: under-specification. Rollback: delete the templates file.
2. **proposal:** Add an `intent_card_version` field to the Worker Issue
   Contract template so any conversational surface can emit cards that
   evolve alongside the contract. Risk: schema drift between intent-card and
   live Worker Issue Contract. Rollback: revert template change.
3. **proposal:** Promote the Source Map shape (Section 8) into a generic
   "Evidence Map" template at `docs/templates/evidence-map.md` so future
   plan artifacts must commit each claim to FACT / INFERENCE / HYPOTHESIS
   in the same row format. Risk: noisy maps in small artifacts. Rollback:
   keep the shape inline and skip the template.
4. **proposal:** When a contract names a `source_of_truth` file that does
   not exist yet, the Contract Controller should emit a soft
   `controller.contract-source-missing` advisory rather than silently
   accepting the path. This run flagged
   `docs/releases/0.6-growth-beta-entry-plan.md` as `HYPOTHESIS`; a soft
   advisory at Stage 0.5 would catch it before worker dispatch. Risk:
   false positives on intentionally forward-looking paths. Rollback:
   gate the advisory behind a `--strict-source-of-truth` flag.
