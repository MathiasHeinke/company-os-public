# 0.7.1 Client Self-Onboarding Readiness

Status: release candidate evidence
Date: 2026-05-26
Owner: CEO / role:coo
Use for: making Company.OS installable, onboardable and updateable for a first
paid non-founder pilot without private sidecars or founder-only paths.

## Goal

Company.OS `0.7.1` / `0.7.2` must prove that a new company can start from the
public/productizable repo and reach a bounded working operating layer:

```text
clone Company.OS
-> run install
-> connect required apps
-> complete company onboarding
-> create first Plane parent + worker contracts
-> run read-only / manual dispatch pilot
-> receive update notice
-> apply a controlled Company.OS update
```

This is the missing step between internal ARES/Fyn Labs operation and a paid
client install.

## Release Meaning

`0.7.0` is the internal read-only Command Center and EVE/AionUI/Hermes entry.

`0.7.1` is the self-serve candidate baseline:

- portable kit install
- required app stack documented
- clean first-run checks
- no private path assumptions
- no private sidecar content in the repo
- first onboarding packet produced
- combined install/onboarding/update smoke proven

`0.7.2` is the guided pilot baseline:

- end-to-end onboarding routine
- first department lane created
- update routine proven
- operator handoff written
- EVE Chief-of-Staff onboarding queue defined
- first paid non-founder pilot is safe to run with manual gates

`0.7.3` is the local EVE runtime bootstrap baseline:

- Command EVE runtime soul generated into the Hermes sidecar
- AionUI-local `hermes` shim verified
- private Hermes default model configured and smoke-tested
- Hermes one-shot EVE smoke report written
- AionUI WebUI start command or detached start path verified
- no Plane write token, production secret or browser credential exposed to
  AionUI/Hermes

Self-serve unsupported public release remains blocked until the productization
readiness gate explicitly passes in public mode.

## Non-Goals

- no unsupervised client self-serve release
- no autonomous Plane Done
- no default scheduler autonomy
- no production DB/schema/RLS/auth writes during first install
- no public publishing, outreach, spend or customer communication by default
- no raw private memory, founder chat logs, local sidecars or credentials in
  Company.OS
- no promise that every client uses every supported app

## Required Stack Definition

Canonical registry:

## Install Routine

The install routine must be executable and boring.

Minimum command shape:

```bash
node scripts/install/bootstrap.mjs install \
  --source ${COMPANY_OS_ROOT} \
  --target ${CLIENT_WORKSPACE_ROOT}
```

`0.7.1` adds a higher-level installer or guided command that:

1. checks Node/Bun/Git availability
2. checks target workspace safety
3. copies `kits/company-os-kit/`
4. writes `.company-os/install-record.md`
5. writes or validates `.company-os/env.example`
6. writes `.company-os/operations/workspace-registry.json`
7. writes `.company-os/operations/software-stack.md`
8. writes `.company-os/operations/human-gates.md`
9. writes `.company-os/company-discovery-brief.md`
10. writes `.company-os/first-run-checklist.md`
11. refuses to overwrite existing agent rules or local state unless `--force` is explicit
12. emits a machine-readable install result with `files_to_copy`,
    `generated_files`, `files_generated`, `collisions` and `next_steps`

Current bootstrap implementation:

```text
scripts/install/bootstrap.mjs
scripts/install/bootstrap-core.mjs
scripts/install/bootstrap.test.mjs
kits/company-os-kit/.company-os/env.example
kits/company-os-kit/.company-os/operations/*.example.md
```

```text
scripts/install/self-serve-smoke.mjs
scripts/install/self-serve-smoke-core.mjs
scripts/install/self-serve-smoke.test.mjs
```

Use it before claiming a new install/update baseline:

```bash
node scripts/install/self-serve-smoke.mjs run \
  --source ${COMPANY_OS_ROOT} \
```

The smoke creates or uses a fresh target, runs bootstrap dry-run, bootstrap
apply, first-company-packet materialization from the installed intake example,
update check report writing and update apply dry-run. It is an evidence drill,
not an unattended client installer.

## Onboarding Routine

Onboarding starts after technical install.

Required outputs:

- `.company-os/install-record.md`
- `.company-os/company-discovery-brief.md`
- `.company-os/onboarding/intake-record.json`
- `.company-os/onboarding/eve-boot-packet.json`
- `.company-os/onboarding/first-plane-parent-draft.md`
- `.company-os/operations/workspace-registry.json`
- `.company-os/operations/software-stack.md`
- `.company-os/operations/human-gates.md`
- `.company-os/first-run-checklist.md`
- `reports/company-discovery/YYYY-MM-DD/business-pressure-test.md`
- `reports/company-discovery/YYYY-MM-DD/first-company-packet.md`
- first Plane parent work item
- first 3-7 child worker contracts with `dispatch: manual`
- first operator handoff

Current first-packet implementation:

```text
scripts/onboarding/first-company-packet.mjs
scripts/onboarding/first-company-packet-core.mjs
scripts/onboarding/first-company-packet-core.test.mjs
kits/company-os-kit/.company-os/onboarding/company-intake.example.json
kits/company-os-kit/.company-os/onboarding/eve-boot-packet.example.json
```

The script consumes a confirmed intake JSON, writes the company discovery brief,
records a file-backed intake, records the EVE onboarding setup queue, and
renders the first Plane parent/child contract drafts without writing to Plane
or enabling dispatch.

The first intake must give EVE enough signal to behave like Chief-of-Staff
beside the Founder, not like a generic assistant. It records:

- existing ledgers, tools, task sources, roles and people sources to review
  before creating new Company.OS structure
- whether the client already uses Plane, Linear, GitNexus, Honcho, GitHub,
  Drive, CRM, analytics or another execution/knowledge surface
- the adoption policy for existing work: adapt existing systems first,
  read-only inventory before migration, and map tasks/roles before duplicates
- founder decision style, communication notes, preferences, refusal patterns
  and taste notes
- accounts/connectors to connect, websites EVE should guide the operator to
  open, permissions needed and tools to install
- requested skills/workflows, including future department-specific asks such as
  a CMO UGC video pipeline
- first goals that should become Plane parent/child drafts
- allowed and forbidden memory sources, with memory saves requiring explicit
  confirmation by default

The generated EVE boot packet is the runtime-facing compression of that intake.
It prevents a fresh founder install from feeling like a blank chatbot:

```text
.company-os/onboarding/eve-boot-packet.json
```

Source doctrine:

```text
docs/operations/eve-first-run-founder-onboarding.md
```

The boot packet separates account seed, runtime probe, capability tiers, first
greeting protocol, progressive setup queue, Company.OS-native core
requirements, optional connectors, memory boundaries and existing-system
adoption policy.

Required first-run order:

1. identify company, domain, offer, buyer and founder bottleneck
2. classify data sensitivity and HumanGate owners
3. choose the first department wedge
4. inventory existing systems: execution ledger, open tasks, roles, people,
   knowledge base, code graph, memory, integrations and current workflows
5. map existing work into Company.OS concepts before creating new tasks,
   parents or child contracts
6. build EVE's setup queue: account pages, permission asks, tool installs,
   memory boundaries, skills/workflows and first goals
7. connect only the apps needed for that wedge
8. create the first Plane parent only after existing task sources are mapped
9. generate child contracts
10. run read-only / manual worker smoke
11. create a morning/reporting rhythm
12. decide whether the pilot may proceed beyond L2

EVE may guide the operator through setup actions such as opening Plane, Honcho,
GitNexus, GitHub or department-tool pages and asking for the missing approvals.
EVE must not create accounts, approve spend, persist private memory, install
unreviewed skills or dispatch workers without the relevant HumanGate and
CEO/Codex review.

If the client already uses Linear, Jira, Notion, Trello, Plane or another
ledger, EVE must not assume a blank slate. EVE first asks for read-only
visibility, summarizes what already exists, identifies roles/owners and maps
existing tasks into Company.OS parent/child concepts. Migration or duplication
requires CEO/Codex review and remains manual-gated.

## Update Routine

Company.OS needs an explicit update command before a paid pilot.

Target command language:

```text
/update_eve
company-os update check
company-os update apply --to 0.7.2 --dry-run
company-os update apply --to 0.7.2
```

Current local implementation:

```text
scripts/update/company-os-update.mjs
scripts/update/company-os-update-core.mjs
scripts/update/company-os-update-core.test.mjs
```

Use it as:

```bash
node scripts/update/company-os-update.mjs check \
  --source ${COMPANY_OS_ROOT} \
  --target ${CLIENT_WORKSPACE_ROOT} \
  --to 0.7.2 \
  --write-report

node scripts/update/company-os-update.mjs apply \
  --source ${COMPANY_OS_ROOT} \
  --target ${CLIENT_WORKSPACE_ROOT} \
  --to 0.7.2 \
  --dry-run
```

The first implementation is a local script, not a full CLI. It must:

- detect current installed Company.OS version
- fetch or compare the source repo version
- list changed kit files, docs, templates and scripts
- refuse to overwrite client-local files silently
- write an update report
- preserve install-specific `.company-os/` local state
- notify the operator what changed and what requires manual review
- keep EVE/Hermes/AionUI adapter updates behind explicit confirmation

`/update_eve` means: run `check`, show the report, ask the operator for
approval, then run `apply --dry-run`; only after approval should the actual
apply command run.

## Acceptance Criteria

`0.7.1` can be claimed when:

- productization readiness guided gate passes
- a fresh target install passes from current repo
- self-serve smoke passes for install, onboarding packet and update dry-run
- required stack registry exists and is cited from install docs
- install routine produces a versioned install record
- no hard-coded private paths are required for the install path
- first onboarding checklist is generated
- Plane parent + child contract template is ready for a new company
- update routine has at least a dry-run checker

`0.7.2` can be claimed when:

- one non-founder pilot can be installed and onboarded with operator guidance
- update routine can apply a kit/doc/template update without clobbering local
  client state
- AionUI/Hermes/EVE pilot path is reproducible from public docs plus local
  sidecar bootstrap
- EVE's Soul/Chief-of-Staff onboarding contract is loaded before the pilot
  smoke and the generated first-company packet includes an EVE setup queue plus
  `.company-os/onboarding/eve-boot-packet.json`
- first initialization includes existing-system discovery and adapts to
  pre-existing ledgers/tasks/roles instead of forcing a fresh-start install
- first department wedge can produce a read-only report through Plane
- HumanGate owner sees a clear approve/hold/park decision surface
- no production writes, external sends, publishing or spend happen by default

`0.7.3` can be claimed when:

- `scripts/operator-shell/start_eve.mjs check --apply-overlay --auth-check`
  passes and writes a `start-eve-preflight` report
- `scripts/operator-shell/eve-sidecar.mjs prepare` writes Command EVE's Hermes
  `SOUL.md`, private overlays and local shims
- `scripts/operator-shell/eve-sidecar.mjs preflight` passes against the local
  AionUI/Hermes sidecars
- `scripts/operator-shell/eve-sidecar.mjs smoke --write-report` writes
  `reports/operator-shell/YYYY-MM-DD/eve-aionui-hermes-start-packet.*`
  with non-empty EVE output; an empty zero-exit Hermes response is a blocker
- `scripts/operator-shell/eve-sidecar.mjs start --port 25809 --write-report`
  starts AionUI WebUI locally without exposing write tokens or secrets
- EVE introduces itself as Command EVE, says what it already knows from the
  boot packet before asking for more, starts the existing-system inventory,
  drafts Founder Intent and CEO Delegation packets and refuses direct
  execution/Done/HG-4
