## 1. Purpose And Non-Goals

The 0.6B Blog Workflow Approval Packet is a **text-first publication-ready
workflow specification**. It defines the seven-stage pipeline a blog post
moves through inside Company.OS, who owns each stage, what gate each stage
must pass, what artifact each stage produces, and how an idea becomes a
parseable Plane child Worker Contract. Publication remains explicitly gated:
nothing in this spec authorizes auto-publish, scheduling, outreach, customer
data access, ad spend or any production write.

It is explicitly NOT:

- a publishing engine, CMS adapter or CDN integration
- a scheduling layer (Buffer / Hypefury / Make / WordPress Cron)
- an analytics pipeline or experiment platform
- a customer-data-aware personalization surface
- a department pack (those land in `0.8.x-beta` per
  `docs/strategy/autonomy-product-horizon.md`)
- authorization for HG-2.5 release of any blog artifact — releases remain
  CEO/Founder owned per `docs/governance/ceo-release-authority.md`

`FACT(docs/strategy/autonomy-product-horizon.md)` lines 110–137 place the
text-first conversation map and its sibling wedge artifacts at `0.6.x-beta`
step 1 and explicitly reserve voice, read-only Command Center POC and
department-pack publishing for `0.7.x-beta` and `0.8.x-beta`.

`FACT(docs/releases/0.6-growth-beta-entry-plan.md)` lines 29–36 list the blog

## 2. Scope Boundaries (Include / Exclude)

Included:

- seven-stage blog pipeline: `idea` -> `outline` -> `draft` -> `editorial` ->
  `claim_review` -> `staged_publish` -> `review_queue`
- per-stage owner, gate, artifact, allowed write path and exit criterion
- idea-to-Plane-child conversion rule that emits a parseable Worker Issue
  Contract per `docs/templates/worker-issue-contract.md`
- editorial review boundary (mandatory before any public-facing artifact may
  move past stage 4) and claim review boundary (mandatory before any
  regulated-claim artifact may move past stage 5)
- explicit `no-public-publish-from-chat` rule and refuse-with-redirect
  template for the conversational surface
- HumanGate map across all seven stages with HumanGateOwner per stage
- evidence-and-source map for every load-bearing claim in this artifact

Excluded:

- live posting on social, blog, newsletter, email, or third-party platforms
- scheduling external posts (Buffer / Hypefury / Make / cron / CMS scheduler)
- outreach sends, distribution lists, customer email
- regulated-claim approval (medical, legal, Rx, financial outcome claims)
- spend changes, ad budget changes, ad copy approvals
- production deployment of any kind, including landing-page deploys that
  embed the published post
- direct Plane state mutation by the worker (no `Done`, no status flips, no
  comment writes other than the gated worker.reported and structured events)
- CMS-specific adapters; this spec is platform-agnostic until a separate
  contract names a CMS

## 3. Pipeline Map: Seven Stages

The blog pipeline is a directed acyclic flow of seven stages. Each stage has
a single owner, a single gate, at least one artifact, an allowed write path,
and a stable exit criterion that the next stage's owner can verify without
re-running the previous stage.

```text
idea -> outline -> draft -> editorial -> claim_review -> staged_publish -> review_queue
```

### 3.2 Stage Notes

- `<slug>` is the lowercased kebab-case of the idea title, deterministic and
  reused across all seven stage artifacts so the audit trail is one folder.
- `reports/marketing-drafts/<slug>/` is a **proposed** allowed write path for
  the downstream child Worker Contracts; this 0.6B artifact does not create
  any draft, so the path is named here for spec completeness and is not
  written by this run.
- The owner of S4 (`editorial`) MUST be different from the owner of S3
  (`draft`). A worker may not editorially review its own draft. If only one
  agent is available, S4 routes to a human reviewer.
- S5 (`claim_review`) ALWAYS runs, even if the draft contains no regulated
  claims, because the claim-review stage is also the freshness/evidence
  audit. The output may be a "no regulated claims, freshness PASS" one-liner;
  it must still be produced.
- S6 (`staged_publish`) targets internal staging only. There is no S6 path
  that touches a public URL. A separate, explicitly contracted worker handles
  public publication after S7 promotion, and that worker is out of scope for
  0.6B.
- The concrete S6 staging surface is intentionally named by the S6 child
  Worker Contract; 0.6B defines the approval packet shape, not the preview
  adapter.

## 4. Idea-to-Plane-Child Conversion Rule

An idea becomes a Plane child work item via a single deterministic transform.
The conversational surface (per 0.6A Section 6) emits an **intent card**.
A human or `role:cmo` seat reviews and pastes the card into Plane. The
dispatcher's parser at `scripts/orchestration/worker-ledger-validator.mjs`
then accepts or rejects the resulting Plane work item.

The intent card MUST be a fenced YAML block matching the flat snake_case
shape required by `docs/templates/worker-issue-contract.md` Plane Dispatcher
Parseability Rule. The conversational surface MUST NEVER write directly into
Plane.

## 5. Editorial Review Boundary

The editorial gate at S4 is the **last review before any artifact may move
toward a public-facing surface**. Crossing it triggers two stable
invariants:

1. **Owner separation.** The editorial reviewer at S4 MUST be a different
   entity from the writer at S3. A single agent may not produce and approve
   its own draft. When only one agent is available, the editorial owner is
   the human `role:cmo` seat or a separate controller agent under a
   different capability profile.
2. **Re-review on edit.** Any change to the draft after S4 PASS invalidates
   the editorial review. A new S4 review MUST run before S5 can lock. The
   audit trail records the editorial-pass hash and the post-edit draft hash
   so the controller can verify the diff.

Editorial gate scoring dimensions (rubric stored in
`reports/marketing-drafts/<slug>/04-editorial-review.md`):

- **Voice fit.** Matches local persona files and voice guide if present.
- **Structure.** Headline -> subheads -> closing flow; no orphan claims.
- **Freshness.** All facts and statistics have a date within the freshness
  window declared by the draft, or are explicitly marked evergreen.
- **Source quality.** Every claim cites a path or URL; no Honcho-only
  "remembered fact" citations.
- **Persona fit.** Speaks to the persona named in the intent card and the
  outline.

If any dimension scores REJECT, S4 returns the draft to S3 with a list of
required edits. There is no S4 override that lets a draft skip to S5.

## 6. Claim Review Boundary

The claim gate at S5 always runs. It exists for two reasons:

1. **Regulated claim safety.** Medical, legal, Rx, financial outcome,
   security and any other regulated claim must be routed to the Claim /
   Compliance Gate child workflow (specified separately as 0.6D per
   `docs/releases/0.6-growth-beta-entry-plan.md` Suggested Child Contracts
   line 4). The blog workflow does not approve regulated claims; it
   identifies them and routes them. Regulated-claim approval is HG-4 and
   Founder-owned per `docs/templates/worker-issue-contract.md` HG-4 rule.
2. **Freshness and evidence audit.** Even non-regulated copy benefits from a
   second check on freshness and source quality. The output is a short
   `05-claim-review.md` artifact that names each claim, its evidence path,
   and its verdict.

When a regulated claim is detected:

- S5 produces `05-claim-review.md` with `verdict: REGULATED_CLAIM_DETECTED`.
- The blog workflow PAUSES at S5; the draft cannot proceed to S6 until the
  Claim / Compliance Gate child workflow (0.6D) returns either
  `claim_approved` from the Founder (HG-4) or `claim_removed` from the
  writer.
- The conversational surface MUST refuse any chat request that asks to push
  the regulated draft past S5 without HG-4, using the refuse-with-redirect
  template from `docs/releases/0.6a-marketing-growth-command-center-map.md`
  Section 6.3.

## 7. No-Public-Publish-From-Chat Rule

**The conversational surface MUST NEVER cause a public publication.**

This rule binds the entire blog workflow:

- Stages S1 through S5 produce only internal artifacts under
  `reports/marketing-drafts/<slug>/`. They never touch a public URL, a CMS
  publish endpoint, a scheduling tool or a distribution list.
- Stage S6 (`staged_publish`) targets internal staging only. Even when the
  CEO releases the HG-2.5 release card for S6, the action is restricted to
  internal preview surfaces (staging URL, draft preview, internal Slack-style
  drop). No public URL is generated by S6.
- Stage S7 (`review_queue`) is post-publish review — but the actual public
  publish is performed by a separate worker explicitly contracted for the
  destination platform (CMS, mirror, newsletter, etc.). That worker is out
  of scope for the 0.6B spec.
- The conversational surface refuses with this template if asked to publish:

  > "I cannot publish blog content from chat. Public publishing requires
  > HG-2.5 release approval from CEO/Founder per
  > `docs/governance/ceo-release-authority.md`, and the actual publish is
  > performed by a separately contracted publish worker, not by this
  > surface. Routing this draft to Stage S6 (`staged_publish`) intent card
  > for CEO release review."

- The conversational surface refuses with this template if asked to schedule
  a publish:

  > "I cannot schedule external publishing from chat. External scheduling is
  > out of scope for 0.6B. Routing this request to the 0.6C Social Approval
  > Queue spec, which defines the scheduling intent card and approval path."

`FACT(docs/releases/0.6-growth-beta-entry-plan.md)` line 32 — the blog

## 10. Reflection

`reflection:`

- The plan-only mode mapped cleanly to the artifact: this document defines a
  seven-stage workflow with owners, gates, artifacts and HumanGate routing,
  but it neither dispatches workers, creates drafts, publishes content,
  schedules external posts, nor mutates Plane state.
- The hardest discipline was holding the line on Stage S6. It was tempting
  to define a public-publish stage to "complete" the pipeline, but the
  0.6B contract scope explicitly excludes auto-publish per the entry-plan
  hard boundary. S6 was kept as internal staging only, and the actual public
  publish was placed out of scope for a separately contracted worker.
- The Stage S5 always-runs rule turned a single regulated-claim guard into a
  general freshness/evidence audit, which keeps the rule simple to operate
  ("S5 always produces an artifact") rather than carving conditional paths
  that operators would have to remember.
- The editorial owner-separation invariant (Section 5) was the second-biggest
  hold-the-line moment. It would have been easier to allow a single agent to
  produce and approve its own draft when only one agent is available, but
  that erodes the audit boundary. The spec falls back to the human `role:cmo`
  seat when only one agent is available, consistent with
  `AGENTS.md` Performance Doctrine "an agent can self-review, but cannot
  self-promote".
- One source-of-truth path declared by the upstream Plane contract still
  ends with a trailing slash for the `reports/releases/0.6.0-beta/`
  directory; this run's report is the first `0.6b-*` artifact in that
  directory. No fabrication of upstream evidence was performed; every
  load-bearing claim is in the Section 9 Evidence Map.

## 11. Learning Proposals

`learning_proposals:` (proposal-only; no durable memory, SOP, skill,
workflow or harness change happens without controller/CEO review)

1. **proposal:** Extract `reports/marketing-drafts/<slug>/` as the canonical
   per-blog audit folder shape and document it under
   `docs/templates/blog-draft-folder-shape.md` so every downstream blog
   worker uses the same `01-idea-card.md` ... `07-review-queue.md` naming.
   Risk: premature standardization before any draft has actually walked the

   ad-hoc until evidence accumulates.
2. **proposal:** Promote the Section 3.1 stage table shape into a generic
   "Workflow Stage Map" template at `docs/templates/workflow-stage-map.md`
   for any 0.6/0.7/0.8 workflow (social queue, claim gate, analytics loop,

   artifact / HumanGate in the same row format. Risk: noisy maps in small
   workflows. Rollback: keep the shape inline per workflow.
3. **proposal:** Add `intent_card_version`, `proposed_blog_stage`,
   `proposed_blog_slug`, `proposed_blog_title`, `proposed_blog_persona`,
   `proposed_blog_primary_keyword` and `proposed_blog_distribution_intent`
   as documented optional fields in
   `docs/templates/worker-issue-contract.md` Intent-Card section so any
   conversational surface that emits a blog intent card uses the same
   parseable shape. Risk: schema drift between intent-card and live Worker
   Issue Contract. Rollback: revert template change; intent-card extensions
   stay per-workflow.
4. **proposal:** Have the Contract Controller (Stage 0.5) emit a soft
   `controller.contract-stage-owner-collision` advisory when a Plane child
   Worker Contract for `proposed_blog_stage: editorial` declares the same

   false positives when the editorial owner is intentionally the same human.
   Rollback: gate behind a `--strict-stage-owner-separation` flag.
5. **proposal:** When the dispatcher computes per-stage `AllowedWritePaths`
   for a blog stage, derive the `reports/marketing-drafts/<slug>/<NN-stage>.md`
