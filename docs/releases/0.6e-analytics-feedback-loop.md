## 1. Purpose And Non-Goals

The 0.6E Analytics Feedback Loop is a **read-only measurement loop specification**.
It defines which metric classes the Marketing/Growth wedge observes, what data sources
each class reads from, how fresh the data must be, who owns the metric class, what
the worker is allowed to do with each metric, what is blocked, and what evidence must
accompany each metric before an insight may flow downstream into the Blog, Social, or
Claim gates.

The loop closes the Growth beta wedge: the 0.6A–0.6D artifacts define how to plan,
draft, review and gate content; this 0.6E artifact defines how the system observes
what happened after content moves through those gates and feeds findings back into
the next planning cycle.

This spec is explicitly NOT:

- a spend-change surface, an ad-budget tool, or a pricing tool
- a production analytics write, a Vercel analytics mutator, a Supabase analytics
  insert, or any external write to a measurement system
- a customer-data reader — no personally identifiable information, no user-level
  row queries, no list exports
- a publishing engine, scheduler, or outreach tool
- a regulated claim approval surface
- a department pack (those land in `0.8.x-beta` per
  `docs/strategy/autonomy-product-horizon.md`)
- authorization for HG-2.5, HG-3 or HG-4 actions — all spend, publish, and regulated-claim
  decisions remain CEO/Founder-owned per `docs/governance/ceo-release-authority.md`

`FACT(docs/releases/0.6-growth-beta-entry-plan.md)` line 35 — the analytics
feedback loop track has first artifact "Read-only metric ingest and weekly feedback
report" with boundary "No spend changes". This spec enforces that boundary; it
does not widen it.

`FACT(docs/strategy/autonomy-product-horizon.md)` lines 93–95 — the `0.6.x-beta`
Growth beta target is "campaigns draft autonomously, publish gated". Analytics
without spend changes is consistent with this target; no spend signal flows back
to the worker as an actionable budget change.

## 2. Scope Boundaries (Include / Exclude)

Included:

- ten (10) metric classes with stable identifiers (acceptance criterion: ≥8)
- per-class: source, freshness window, owner, allowed action, blocked action,
  evidence requirement
- weekly feedback report schema (pasteable into Plane without external writes)
- insight-to-action handoff rules for Blog (0.6B), Social (0.6C), and Claim (0.6D)
  gates without bypassing editorial or claim review
- HumanGate map for spend, public publishing, customer data, and regulated claims
- evidence and source map for every load-bearing claim
- reflection, learning_proposals, and capability_context_proof

Excluded:

- live spend changes, ad budget changes, or pricing changes
- production analytics writes (Vercel, Supabase, Stripe, Google Analytics mutations)
- customer-level data queries, user-level row reads, PII access, list exports
- external publishing, scheduling, or outreach sends
- regulated claim approval or drafting of regulated copy
- direct Plane state mutation (no Done, no status flips from this spec)
- building a dashboard, a live reporting surface, or a charting system
- detailed A/B test or experiment platform design (deferred to `0.8.x-beta`)

## 3. Metric Classes

### 3.1 Overview

Ten metric classes are defined. Every class has a stable machine-readable identifier,
a human-readable name, the data source the worker may read, the freshness window for
the data to be valid, the metric owner, the allowed action when a metric is consumed
in the feedback loop, the blocked action, and the evidence requirement for any
downstream handoff.

The ten classes satisfy the acceptance criterion of at least eight metric classes with
margin, allowing the controller to prune without re-opening the contract.

## 4. Weekly Feedback Report Schema

### 4.1 Purpose

The weekly feedback report is produced by a `role:cmo` worker reading the ten
metric classes (Section 3) and summarizing findings into a single artifact. It:

1. can be pasted directly into a Plane comment or work-item description without
   any external write
2. maps every quantitative finding to one of four action classes (see §4.3)
3. does not change spend, does not publish content, does not approve claims, and
   does not mutate any Plane state from the report-generation run itself
4. is a read-only artifact — the human operator or CEO decides whether to act on it

### 4.2 Report Artifact Path

```text
reports/releases/0.6.0-beta/0.6e-weekly-feedback-<ISO-8601-week>.md
```

Where `<ISO-8601-week>` is the ISO week identifier (e.g. `2026-W21`). One file
per week. The allowed write path for downstream worker runs that produce weekly
reports is `reports/releases/0.6.0-beta/` (this is within the declared allowed
write path for this work item).

### 4.3 Action Classes

Every finding in the weekly report is tagged with exactly one action class:

| Action Class | Tag | Meaning | Who acts | Gate required |
|---|---|---|---|---|
| Pipeline improvement suggestion | `[IMPROVE]` | A metric signal suggests a draft, outline, editorial, or claim-review step could be improved | `role:cmo` | None — the suggestion is text only; implementation requires a new Plane work item via intent card |
| Human review required | `[HUMAN-REVIEW]` | A metric signal requires a human to inspect or decide (e.g. high stale rate, p90 HG latency spike, REJECT-heavy claim gate) | Human `role:cmo` or CEO | HG-2 at minimum; HG-2.5 for spend or publish signals |
| Founder escalation | `[FOUNDER]` | A metric signal touches spend, customer data, regulated claims, or production systems | Founder (HG-4) | HG-4 mandatory; no worker or CEO may act |
| No action needed | `[OK]` | Metric is within expected range; record for trend baseline | None | None |

### 4.4 Weekly Report Schema (Pasteable Markdown/YAML)

The report is a Markdown file with a YAML front-matter block and a human-readable
body. The front-matter is machine-readable for future dispatcher parsing; the body
is human-readable for the operator and CEO.

## 5. Insight-To-Action Handoff Rules

### 5.1 Principle

An analytics insight that implies a change to Growth content MUST flow through the
existing 0.6B, 0.6C, or 0.6D gates before any content is changed, drafted, or
approved. The analytics loop is read-only; it proposes, it does not execute.

No insight may be used to bypass editorial review (0.6B S4, 0.6C editorial state),
claim review (0.6B S5, 0.6C claim_review, 0.6D gate), or the scheduling intent card
flow (0.6C scheduling_intent). The analytics loop explicitly cannot:

- advance a blog draft to a later stage because a metric looks good
- approve a social post for scheduling because engagement is trending
- remove a regulated claim from a draft based on metric performance
- infer that a claim is defensible because prior similar claims were approved

### 5.2 Handoff Routing Table

| Metric Class | Insight Type | Downstream Gate | Handoff Artifact | Who Acts |
|---|---|---|---|---|
| M1, M2, M4 (content engagement, bounce, conversions) | Blog content improvement needed | **0.6B Blog Workflow** — propose a new S1 `idea` intent card or an S2 `outline` revision intent card | Intent card in YAML per 0.6A §6.2 schema, pasted by human into Plane | `role:cmo` human operator |
| M1, M2, M4 (same metrics for social content) | Social copy improvement needed | **0.6C Social Queue** — propose a new `draft` entry or a revision of a `rejected` draft | Intent card in YAML per 0.6A §6.2 schema | `role:cmo` human operator |
| M3 (traffic source distribution — paid share change) | Spend signal detected | **HG-4 Founder escalation** — no worker or CEO acts; weekly report tags [FOUNDER] | Weekly report section "HumanGate Required Actions" with Founder notification | Founder only |
| M5 (blog draft throughput — stage latency) | Pipeline bottleneck identified | **0.6B Blog Workflow** — propose a pipeline improvement intent card targeting the stalled stage | Intent card targeting the bottleneck stage | `role:cmo` human operator |
| M6 (social queue health — stale posts) | Stale drafts requiring freshness review | **0.6C Social Queue** — stale drafts re-enter `editorial` state per the staleness rule already in 0.6C §4; no new gate needed | Weekly report lists stale slugs; dispatcher fires staleness rule per 0.6C §4 | Dispatcher (automated); human notified |
| M7 (claim gate outcomes — regulated claim unresolved) | Unresolved regulated claim in pipeline | **0.6D Claim/Compliance Gate** — the draft is already paused at the gate; the weekly report confirms the pause and tags [FOUNDER] | Weekly report section "HumanGate Required Actions" | Founder only (HG-4) |
| M7 (claim gate outcomes — high REJECT rate) | Content quality or evidence gap | **0.6B Blog** or **0.6C Social** — propose a content-quality improvement intent card or an evidence-gathering Plane item | Intent card | `role:cmo` human operator |
| M8 (HG release latency — HG-4 > 72h) | Founder queue backlog signal | Weekly report [FOUNDER] tag only — no escalation action by worker or CEO; Founder is informed, not pressured | Weekly report | Founder reads the report |
| M9 (worker run quality — low PASS rate) | Runtime or spec quality issue | CEO review — weekly report [HUMAN-REVIEW] or [FOUNDER] tag; CEO may request a targeted spec improvement item | Weekly report | CEO (or Founder for [FOUNDER] tag) |
| M10 (content evidence freshness — stale regulated source) | Regulated-class source stale | **0.6D Claim/Compliance Gate** — the claim must be re-reviewed before the content item may move forward; weekly report tags [FOUNDER] | Weekly report; existing 0.6D gate runs on the affected draft | `role:cao` + Founder (HG-4) |

### 5.3 What The Handoff Must Not Do

The handoff from analytics insight to content gate MUST NOT:

- use a metric result as justification to skip an editorial review step
- use conversion rate improvement as a proxy for claim defensibility
- propose spend changes through a content improvement intent card (spend changes are
  HG-4 regardless of the framing)
- route a finding directly to the 0.6D claim gate as a claim approval — the analytics
  loop only identifies that an unresolved regulated claim exists; the 0.6D gate is
  already running or must be started through a normal Plane child work item

`FACT(docs/releases/0.6d-claim-compliance-gate.md)` Section 8 — the no-claim-
approval-by-worker rule is absolute; M7 insight-to-action handoffs may not be used to
bypass that rule under any framing.

`FACT(docs/releases/0.6b-blog-workflow-approval-packet.md)` Section 5 — the editorial
owner-separation invariant and the re-review-on-edit invariant are not voidable by
analytics findings; a draft that changes after S4 PASS always needs a new S4 review.

## 6. Blocked Action Catalog (Analytics Surface)

The analytics feedback loop surface MUST refuse the following action classes
regardless of how the request is framed.

| Class | Examples | Authority required to ever execute | Refusal template |
|---|---|---|---|
| Spend change | "Our paid traffic share is up 10% — raise the ad budget", "the ROI looks good, allocate more spend" | HG-4 (Founder) — spend is always Founder-owned | "I cannot change spend. ## 9. Reflection

`reflection:`

- The plan-only mode mapped cleanly: this document defines metric classes, a weekly
  report schema, handoff rules, and a blocked-action catalog without reading any live
  Clarity data, producing any spend change, accessing any customer data, or mutating
  any Plane state.
- The hardest design discipline was the M3 (traffic source distribution) to spend
  signal handoff. It was tempting to define an "if paid share increases materially,
  route to CFO for budget review" path that felt like a workflow improvement. Instead,
  any paid-share signal is tagged [FOUNDER] in the weekly report, and the worker is
  explicitly blocked from framing it as a content-improvement intent card. The
  boundary is total: analytics insights cannot become spend authorizations through
  any framing.
- The M7 (claim gate outcomes) metric class required careful scope. It reads claim
  verdict aggregates from the 0.6D artifacts, but it cannot use the aggregate as
  a claim-approval signal. A 90% PASS rate on the claim gate does not make the 10%
  REGULATED_CLAIM_DETECTED items defensible; those items are still paused at HG-4.
  The spec is explicit about this in §5.3.
- The weekly report schema (§4.4) is deliberately verbose in its YAML front-matter.
  A machine-readable schema for the report is useful for future dispatcher parsing;
  a human-readable body is needed for the operator and CEO. The two are combined
  in one Markdown-with-YAML file so there is only one artifact per week, not two.
- The 0.6B/0.6C/0.6D insight-to-action handoff rules (§5.2) were derived from first
  principles and then cross-checked against the blocked-action catalogs in each prior
  spec. No new gate authority was introduced; the analytics loop is a proposal
  surface, not an execution surface.
- The HumanGate map (Section 7) was designed to be exhaustive rather than minimal.
  A controller reviewing this spec should be able to confirm that every action class
  with real-world side effects is gated and that no analytics finding bypasses any
  prior gate.
- CFO is named as a stakeholder for measurement discipline but does not own the Growth
  wedge. The M3 paid-traffic signal and M9 cost signal are the two places where CFO
  visibility is explicitly flagged. Neither signal grants CFO authority to change spend
  from this loop; that authority remains Founder-owned at HG-4.
- All ten metric class `blocked_action` entries were cross-checked against the 0.6A
  blocked-action catalog and the 0.6D claim gate rules. No inconsistency was found.

## 10. Learning Proposals

`learning_proposals:` (proposal-only; no durable memory, SOP, skill, workflow
or harness change happens without controller/CEO review)

1. **proposal:** Extract the metric-class table shape (§3.2: class ID, name, source,
   freshness window, owner, allowed action, blocked action, evidence requirement) into
   a generic "Metric Class Map" template at `docs/templates/metric-class-map.md` so
   future measurement loops (0.7 morning brief, 0.8 department analytics) use the
   same column format. Risk: premature abstraction before a second measurement loop
   exists. Rollback: delete the template; per-loop metric tables remain per-workflow.

2. **proposal:** Add `weekly_feedback_report_version` as a documented field in
   `docs/templates/worker-issue-contract.md` Intent-Card section (alongside the
   `intent_card_version` and `scheduling_intent_card_version` proposals from 0.6A and
   0.6C) so any conversational surface that generates a weekly analytics report uses
   the same versioned front-matter schema. Risk: schema drift between the 0.6E report
   and the canonical template. Rollback: revert template change; 0.6E report stays a
   per-workflow schema.

3. **proposal:** Have the Runtime Dispatcher emit a weekly `ledger.run_summarized`
   event when the analytics feedback worker completes, carrying the `overall_health`
   field (GREEN/YELLOW/RED) as a structured field in `metrics/agent-events.jsonl`.
   This would allow the morning brief (0.7) to pick up the health signal without
   re-reading the full report artifact. Risk: event-ledger inflation if health changes
   frequently. Rollback: gate behind a `--emit-weekly-health-event` dispatcher flag.

4. **proposal:** Define a `measurement_freshness_policy` doc at
   `docs/governance/measurement-freshness-policy.md` where the Founder explicitly
   approves the freshness windows per metric class (currently: ≤7 days for Clarity
   reads, ≤30 days for claim gate and agent-events aggregates). These windows are
   operational inferences; Founder calibration similar to the 0.6D freshness window
   calibration proposal would make them load-bearing governance rather than inference.
   Risk: governance overhead if windows need frequent adjustment. Rollback: keep
   windows inline in this spec.

5. **proposal:** Cross-reference this analytics loop with the planned 0.6F Editorial
   Controller Rubric (`docs/releases/0.6-growth-beta-entry-plan.md` Suggested Child
   Contracts line 6) so the editorial controller's weekly synthesis pass can consume
   the 0.6E weekly report as one of its input signals. Specifically, the M7 claim
   gate outcome rate and M10 content evidence freshness compliance rate are natural
   inputs for a controller rubric that grades voice, claims, freshness and source
   quality. Risk: coupling the editorial rubric to the analytics loop prematurely
   before 0.6F is defined. Rollback: keep the two specs independent; editorial
   controller computes its own freshness check without consuming the weekly report.

6. **proposal:** Promote the action-class tagging system ([OK], [IMPROVE],
   [HUMAN-REVIEW], [FOUNDER]) from the weekly report schema into a generic
   "Action-Class Tag Library" at `docs/templates/action-class-tags.md` so any
   Company.OS report artifact (morning brief, goal synthesis, harness output) uses
   the same four-tag vocabulary. Risk: tag semantics may diverge between domains
   (e.g. [FOUNDER] in a runtime report has different urgency than [FOUNDER] in an
   analytics report). Rollback: keep tags per-workflow; do not promote to a shared
   library.
