## 1. Purpose And Non-Goals

The 0.6F Editorial Controller Rubric is a **deterministic grading harness** for
Marketing/Growth review packets produced by the `0.6.x-beta` Growth wedge. It
enables a CAO or Codex Deputy-CEO controller to evaluate any blog workflow approval
packet (0.6B), social approval queue entry (0.6C), claim/compliance gate artifact
(0.6D), or analytics feedback report (0.6E) and issue a structured PASS, WARN, or
FAIL verdict with evidence, before any release or publication action.

The rubric defines ten scoring dimensions, the evidence required to PASS or WARN on
each, and the hard-fail conditions that trigger an unconditional FAIL regardless of
other dimension scores. The rubric does not generate content, approve claims, release
HumanGates, publish, schedule, access customer data, or spend budget.

This spec is explicitly NOT:

- a publishing engine, CMS adapter, or distribution tool
- a claim-approval surface — regulated claim approval remains HG-4 (Founder) per
  `docs/releases/0.6d-claim-compliance-gate.md` Section 8 and
  `docs/governance/ceo-release-authority.md`
- a content-generation surface — the rubric grades artifacts produced by CMO/CTO
  workers; it does not produce drafts
- a replacement for the 0.6D Claim/Compliance Gate — the rubric cross-checks that
  the gate ran and returned a valid verdict; it does not re-adjudicate claims
- authorization for HG-2.5, HG-3 or HG-4 actions — all publish, spend, and regulated-claim
  decisions remain CEO/Founder-owned
- a substitute for the human `role:cmo` editorial review at 0.6B S4 or the 0.6C
  `editorial` state — those gates require a different human or agent session from the
  writer; this rubric runs after those gates, not instead of them

`FACT(docs/releases/0.6-growth-beta-entry-plan.md)` — the editorial controller review
track boundary is "CAO does not create content." This spec enforces that boundary; it
defines only the grading surface.

`FACT(docs/agents/cao.md)` — CAO does not build, does not edit, does not approve, does
not transition to Done. This rubric is a CAO-seat artifact; the same constraints apply.

## 2. Scope Boundaries (Include / Exclude)

Included:

- ten (10) scoring dimensions with PASS / WARN / FAIL semantics (acceptance criterion:
  ≥8 dimensions satisfied; this rubric provides 10 with margin)
- per-dimension: description, evidence requirement for PASS, evidence requirement for
  WARN, and hard-fail trigger
- nine (9) hard-fail conditions that force overall FAIL regardless of other scores
- a pasteable controller verdict schema (YAML) for Plane comments
- outcome → next action routing for PASS, WARN, and FAIL verdicts, routing back to
  0.6B / 0.6C / 0.6D / 0.6E where appropriate, without granting Plane Done or
  external-write authority
- HumanGate map for every verdict outcome
- evidence and source map for every load-bearing claim in this artifact
- reflection and learning proposals
- capability context proof

Excluded:

- generating, drafting, editing, or approving any marketing or claim content
- approving regulated claims (medical, legal, Rx, financial, security, privacy) —
  each remains HG-4 regardless of how the rubric scores
- authorizing public publishing, external scheduling, outreach, or spend changes
- reading customer data, user-level analytics, or PII
- direct Plane state mutation (no Done, no status flips, no comment writes other than
  the controller.verdict comment)
- replacing the 0.6B S4 editorial gate or the 0.6C `editorial` state — these must
  run before the rubric runs
- issuing verdicts on work outside the 0.6 Marketing/Growth wedge

## 3. Scoring Dimensions

The rubric grades ten dimensions. Every dimension produces exactly one of:
`PASS`, `WARN`, or `FAIL`. A single hard-fail condition in any dimension causes
the overall verdict to be `FAIL` regardless of other dimension scores.

The overall verdict is:

- `PASS` — all ten dimensions score PASS; no hard-fail conditions triggered
- `WARN` — no FAIL in any dimension; at least one WARN; no hard-fail triggered
- `FAIL` — any dimension scores FAIL, OR any hard-fail condition triggered

### D1 — Voice And Brand Fit

**What is graded:** Whether the artifact's content (drafts, copy, reports) uses the
voice and tone consistent with the persona files and any local voice guide, and whether
it avoids register mismatch relative to the declared audience.

**Evidence required for PASS:**
- A signed `04-editorial-review.md` (blog) or `editorial-review.md` (social) artifact
  is present in the artifact folder.
- The editorial review artifact explicitly records a PASS verdict on the `Voice fit`
  dimension.
- The artifact names the persona file or voice guide against which it was checked
  (path must be in `kits/company-os-kit/.agents/personas/` or a local `voice-guide.md`).

**Evidence for WARN (no FAIL):**
- The editorial review artifact is present and PASS, but the persona file path is not
  named (auditor cannot verify which guide was used).
- Voice guide does not exist in the workspace; reviewer self-assessed against general
  brand norms — note the gap in the verdict.

**Hard-fail trigger:** The editorial review artifact is absent and no human reviewer
signed off on voice fit. The artifact was not editorially reviewed before the rubric
ran.

**Source:** `FACT(docs/releases/0.6b-blog-workflow-approval-packet.md)` Section 5
editorial gate scoring dimension "Voice fit"; Section 3.1 Stage S4 owner-separation
invariant; `FACT(docs/releases/0.6c-social-approval-queue.md)` Section 5 editorial
review dimensions.

### D2 — Source Quality

**What is graded:** Whether every factual claim in the artifact cites a verifiable
path (repo-relative file path, URL, or Plane work item ID) and whether the cited
sources are authoritative rather than Honcho-only memory citations.

**Evidence required for PASS:**
- Every factual claim in the artifact ends with `(source: <path or url>)` or
  `(file: <path>)` or `(plane: <work-item-id>)` inline citation.
- The editorial review artifact explicitly records a PASS verdict on the `Source
  quality` dimension.
- No citation is of the form `(source: memory)` or `(source: honcho)` or any other
  Honcho-read-only reference used as a factual source.

**Evidence for WARN:**
- At least one citation uses a URL rather than a repo-relative path (URL cannot be
  audited against the current repo state; flag for freshness check in D3).
- The editorial review artifact is PASS on source quality but does not explicitly
  enumerate which claims were checked.

**Hard-fail trigger:** At least one factual claim has no citation at all (uncited
fact). An uncited fact is any sentence containing a quantitative assertion, a named
external reference, or a product capability claim that lacks an inline source marker.

**Source:** `FACT(docs/releases/0.6a-marketing-growth-command-center-map.md)` Section 4
Source Citation Rules ("Cite or refuse"); `FACT(docs/releases/0.6b-blog-workflow-approval-packet.md)`
Section 3.1 Stage S3 gate `draft_pass` — "every claim ends with `(source: <path or url>)`";
`INFERENCE(docs/orchestration/claude-clevel-worker-runtime.md)` — Honcho is a context
store, not a factual evidence repository.

### D3 — Evidence Freshness

**What is graded:** Whether the sources cited in the artifact fall within the
applicable freshness windows — per the 0.6D Claim/Compliance Gate evidence requirement
table for regulated-class sources, and within a 0.6E-derived 7-day window for
Clarity/analytics citations.

**Evidence required for PASS:**
- Every non-evergreen source has a publication or observation date that falls within
  its class-specific freshness window (5 years for medical, 12 months for legal/
  security/privacy, 2 years for financial, 7 days for Clarity metrics).
- The claim review artifact (`05-claim-review.md` or `claim-review.md`) records
  `evidence_freshness` for every claim entry; no entry has `verdict: STALE`.
- Evergreen sources are explicitly labeled "evergreen" in the artifact.

**Evidence for WARN:**
- At least one source has a publication date that cannot be determined from the
  artifact (e.g., the URL is cited but no date is given); the claim is not marked
  evergreen. Auditor flags the claim for freshness verification before release.
- All sources are within window but the claim review artifact did not compute
  `evidence_expiry` for certification-type sources (where expiry matters independently
  of publication date).

**Hard-fail trigger:** At least one source cited in the artifact is outside the
applicable freshness window per the 0.6D minimum evidence requirements table
(`docs/releases/0.6d-claim-compliance-gate.md` Section 4), and the claim carrying
that stale source has not been removed or replaced before the rubric runs.

**Source:** `FACT(docs/releases/0.6d-claim-compliance-gate.md)` Section 4 Minimum
Evidence Requirements table (freshness windows per class); `FACT(docs/releases/0.6e-analytics-feedback-loop.md)`
Section 3.2 M10 — content evidence freshness compliance metric (7-day observation
window for Clarity-derived data).

### D4 — Editorial Separation

**What is graded:** Whether the editorial review at Stage S4 (blog) or the `editorial`
state (social) was performed by a different entity from the content writer, enforcing
the owner-separation invariant.

**Evidence required for PASS:**
- The `04-editorial-review.md` (blog) or `editorial-review.md` (social) artifact
  names the reviewer session or reviewer identity.
- The named reviewer is demonstrably different from the writer named in the
  `03-draft.md` (blog) or the `draft` artifact (social). "Demonstrably different"
  means different Plane work item assignee, different agent CapabilityProfile session,
  or explicit human reviewer name.
- The editorial review artifact carries a signed PASS verdict (not a self-review
  note from the draft author).

**Evidence for WARN:**
- The editorial review artifact is present and PASS, but the reviewer identity field
  is absent or set to a generic value (e.g., "reviewer: n/a"); auditor cannot confirm
  separation.
- Only one agent or operator was available; a human `role:cmo` seat performed the
  editorial review manually — this is allowed per 0.6B Section 5 but must be
  documented.

**Hard-fail trigger:** The writer and editorial reviewer are the same entity (same
Plane assignee, same CapabilityProfile session ID, or same human operator signing
both the draft and the editorial review). Self-review of a draft is explicitly
prohibited per the 0.6B owner-separation invariant; it is an unconditional hard fail.

**Source:** `FACT(docs/releases/0.6b-blog-workflow-approval-packet.md)` Section 5
owner-separation invariant — "The editorial reviewer at S4 MUST be a different entity
from the writer at S3"; `FACT(docs/releases/0.6c-social-approval-queue.md)` Section 5
editorial review — "must differ from writer"; `FACT(AGENTS.md)` Performance Doctrine
lines 159–168 — "an agent can self-review, but cannot self-promote."

### D5 — Claim Gate Compliance

**What is graded:** Whether the claim review gate (0.6D) was run, produced a valid
claim-review artifact, and returned a non-blocking verdict before the artifact moved
past the claim gate stage.

**Evidence required for PASS:**
- A claim-review artifact (`05-claim-review.md` for blog, `claim-review.md` for
  social) exists at the canonical path (`reports/marketing-drafts/<slug>/05-claim-review.md`
  or `reports/social-drafts/<slug>/claim-review.md`).
- The artifact's `overall_verdict` is one of: `PASS`, `REGULATED_CLAIM_APPROVED`,
  or `CLAIM_REMOVED`.
- The artifact's `hg3_escalation_required` is `false` (or `true` only if accompanied
  by `hg4_founder_approved: <comment ID>`).
- Every per-claim entry in `claims_reviewed` has a non-empty `evidence_path` or
  carries `verdict: CLAIM_REMOVED`.
- The claim-review artifact is stamped with `gate_version: 0.6d`.

**Evidence for WARN:**
- The claim-review artifact is present and `overall_verdict: PASS`, but one or more
  per-claim entries lack a publication date for the cited evidence (freshness cannot
  be independently confirmed; flag for D3 cross-check).
- The artifact was produced by the CMO seat alone for ordinary copy without a `role:cao`
  co-reviewer session — allowed only for standard claims without any regulated-class
  detection; flagged because the 0.6D spec nominates `role:cao` as primary auditor.

**Hard-fail trigger (any of the following):**
1. The claim-review artifact is absent — the claim gate was not run.
2. The artifact's `overall_verdict` is `REGULATED_CLAIM_DETECTED` with
   `hg3_escalation_required: true` and `hg4_founder_approved: false` — the regulated
   claim is unresolved and the draft was advanced past the gate anyway.
3. Any per-claim entry carries `verdict: PASS` with an empty `evidence_path`.
4. The `overall_verdict` is `STALE` or `UNSUPPORTED` — the gate identified a
   deficiency that was not resolved before the draft advanced.

**Source:** `FACT(docs/releases/0.6d-claim-compliance-gate.md)` Section 6.1
Claim-Review Log YAML; Section 6.3 Log Completeness Rule; Section 8
no-claim-approval-by-worker rule; `FACT(docs/releases/0.6b-blog-workflow-approval-packet.md)`
Section 6 — "S5 always runs."

### D6 — Publish And Schedule Boundary Compliance

**What is graded:** Whether the artifact (and its producing worker run) respected the
absolute prohibition on public publishing, external scheduling, and autonomous social
platform API writes.

**Evidence required for PASS:**
- The worker.reported for the producing run declares `mode: plan | report | review` —
  no implementation mode that could have triggered a publish.
- The producing run's `blockedactions` field lists "no public posting", "no scheduling
  external posts", "no production deployment", and "no autonomous external scheduling."
- The `reports/releases/0.6.0-beta/` report artifact contains no public URL that
  resolves to a live external page.
- No `human_gate.released` event exists for this item at HG-2.5, HG-3 or HG-4 — the draft
  was not released through any publish gate during this run.

**Evidence for WARN:**
- The worker.reported contains a learning_proposal that suggests adding a publish step
  in the future; this is acceptable (proposals are not actions) but the auditor flags
  it to confirm the proposal is in the `learning_proposals:` block and not in the
  `worker.reported` state field.

**Hard-fail trigger (any of the following):**
1. The report artifact contains a public URL pointing to live external content
   (blog post URL, social platform URL, scheduling tool confirmation) that did not
   exist before this run.
2. The producing worker's tool log (if available) shows a call to an external
   publishing API, scheduling service, or social platform API.
3. The worker.reported `mode` is `implement` or `deploy` on a work item whose scope
   includes content publication.
4. Any external scheduling confirmation is present in the run artifacts.

**Source:** `FACT(docs/releases/0.6-growth-beta-entry-plan.md)` Entry Conditions
lines 22–26 — "No worker can publish, schedule external posts, send outreach, change
spend, make regulated claims or mutate production systems without a human gate";
`FACT(docs/releases/0.6a-marketing-growth-command-center-map.md)` Section 5 Blocked
Action Catalog; `FACT(docs/releases/0.6c-social-approval-queue.md)` Section 8
no-autonomous-external-scheduling rule.

### D7 — Customer Data And Spend Boundary Compliance

**What is graded:** Whether the artifact and its producing run respected the absolute
prohibitions on customer data access and spend changes.

**Evidence required for PASS:**
- The worker.reported and report artifact contain no reference to user-level data,
  PII, customer export, or list access.
- The worker's declared `allowed_connectors` (from its CapabilityProfile) does not
  include any customer-data connector (Supabase row-level queries, CRM, email list).
- The work item's `MaxSpend` is declared and observed (EUR 0 for 0.6 Growth wedge
  work); no `cost_ledger` row shows spend exceeding the declared maximum.
- No Stripe, ad platform, or budget-management tool write appears in the run artifacts.

**Evidence for WARN:**
- The report artifact reads an aggregate analytics metric (M3 traffic source
  distribution from 0.6E) and the paid-traffic percentage changed; the report
  correctly tags this `[FOUNDER]` rather than proposing a spend change — this is
  compliant but the auditor confirms the tag is present.

**Hard-fail trigger (any of the following):**
1. The report artifact contains a user-level row, individual session record, named
   customer, or exported list of any kind.
2. The work item's observed spend exceeds its declared `MaxSpend`, or a spend change
   was proposed through a content-improvement intent card rather than a [FOUNDER]-tagged
   weekly report finding.
3. The producing worker's tool log shows a write to any ad platform, pricing system,
   billing system, or Stripe API.
4. The report artifact proposes a budget change without tagging it `[FOUNDER]` and
   without routing it to Founder at HG-4.

**Source:** `FACT(docs/releases/0.6-growth-beta-entry-plan.md)` Entry Conditions —
"No worker can... change spend... without a human gate"; `FACT(docs/releases/0.6e-analytics-feedback-loop.md)`
Section 6 Blocked Action Catalog — spend change is HG-4 (Founder) with no analytics
bypass; `FACT(docs/governance/ceo-release-authority.md)` — material spend remains
HG-4.

### D8 — Analytics Feedback Loop Compliance

**What is graded:** Whether any analytics-derived insight cited in the artifact was
consumed compliantly — read-only, aggregate-only, with proper action-class tagging,
and without being used to bypass editorial or claim review.

**Evidence required for PASS (applies when artifact cites analytics data):**
- Every Clarity metric citation names a Clarity view ID and a date range.
- Every metric finding in the artifact carries one of the four action-class tags
  (`[OK]`, `[IMPROVE]`, `[HUMAN-REVIEW]`, `[FOUNDER]`) per the 0.6E tagging system.
- No analytics finding was used to advance a blog stage or social queue state without
  going through the appropriate 0.6B or 0.6C gate.
- No analytics finding was used as a claim-defensibility proxy (a high engagement rate
  does not constitute evidence for a regulated claim).

**Evidence for PASS when artifact cites no analytics data:**
- Controller notes that M1–M10 data was not required for this artifact type (e.g., a
  claim-review artifact typically does not cite engagement metrics); mark D8 as
  `PASS (n/a)` with a note.

**Evidence for WARN:**
- Analytics data is cited without a date range or Clarity view ID; the claim is
  otherwise properly tagged. Flag for D3 freshness check.
- An `[IMPROVE]` tagged finding was acted on within the same run rather than converted
  to an intent card for a separate worker — only valid if the action was a text
  revision within the artifact's declared scope.

**Hard-fail trigger:**
- An analytics metric was used to assert that a claim is defensible, to approve a
  social post without an editorial review, or to propose a spend change through any
  channel other than a [FOUNDER]-tagged weekly report finding.
- An analytics write (Clarity goal mutation, Supabase insert, Vercel project write)
  is present in the run artifacts.

**Source:** `FACT(docs/releases/0.6e-analytics-feedback-loop.md)` Section 5.3 — "The
handoff from analytics insight to content gate MUST NOT use a metric result as
justification to skip an editorial review step" and "MUST NOT use conversion rate
improvement as a proxy for claim defensibility"; Section 6 Blocked Action Catalog.

### D9 — Fact Citation Completeness

**What is graded:** Whether every quantitative assertion, named external reference, and
product capability claim in the artifact carries a machine-parseable inline citation.
This dimension operationalizes the "cite or refuse" rule from 0.6A and the `FACT /
INFERENCE / HYPOTHESIS` doctrine from the global worker doctrine.

**Evidence required for PASS:**
- Every paragraph that makes a factual assertion (numeric, comparative, named entity,
  or product capability) ends with at least one inline citation: `(source: <path>)`,
  `(file: <path>)`, `(plane: <work-item-id>)`, `(clarity: <view-id>)`, or uses the
  `FACT(path)` / `INFERENCE(path)` / `HYPOTHESIS` marker from the global doctrine.
- Claims marked `HYPOTHESIS` are explicitly flagged as having no current file evidence;
  the controller accepts HYPOTHESIS as a valid marker (not a citation) only when the
  claim is clearly labeled and the artifact does not treat the hypothesis as a FACT.
- The artifact's Evidence And Source Map (or equivalent) lists every load-bearing claim
  with its evidence class and source.

**Evidence for WARN:**
- One or more supporting (non-load-bearing) claims lack inline citations but are
  present in the Evidence And Source Map with `INFERENCE` class and a cited file.
  Controller accepts INFERENCE as sufficient for non-quantitative contextual claims.

**Hard-fail trigger:** At least one load-bearing claim — defined as a quantitative
assertion, a named external product capability, a compliance status claim, or a claim
that directly supports a recommendation or gate verdict — has no citation, no
evidence-class marker, and is absent from the Evidence And Source Map.

**Source:** `FACT(docs/releases/0.6a-marketing-growth-command-center-map.md)` Section 4
Source Citation Rules rule 1 ("Cite or refuse"); `FACT(AGENTS.md)` global worker
doctrine — "Architektur- und Produktbehauptungen muessen als FACT(file:path),
INFERENCE(file:path) oder HYPOTHESIS(no evidence yet) markiert werden."

### D10 — Worker Report Completeness

**What is graded:** Whether the producing worker's final report includes all three
mandatory structural blocks — `reflection:`, `learning_proposals:`, and
`capability_context_proof:` — as required by the CAO Claude C-Level Worker Pass and
the dispatcher v1.2+ reporting contract.

**Evidence required for PASS:**
- The report artifact (or worker.reported Plane comment) contains a literal
  `reflection:` block with at least two substantive sentences describing what the
  worker did, what tensions arose, and what the hardest discipline point was.
- The report artifact contains a literal `learning_proposals:` block with at least
  one numbered proposal or an explicit statement that no learning was found (with
  reason).
- The report artifact contains a literal `capability_context_proof:` block naming the
  CapabilityProfile loaded, the plugins and commands consumed, and an explicit
  confirmation that no forbidden surfaces were used.
- The worker.reported Plane comment emits an explicit `subagents: []` block when
  `SubAgentRoster` is empty, per `docs/orchestration/subagent-reporting-contract.md`.

**Evidence for WARN:**
- All three blocks are present but the `reflection:` block is a single sentence
  without substantive content (e.g., "Work completed successfully"). Controller
  accepts the block but flags shallow reflection.
- The `capability_context_proof:` block names the profile but does not list the
  specific plugins or commands consumed.

**Hard-fail trigger (any of the following):**
1. The `reflection:` block is absent from the report artifact and the Plane comment.
2. The `learning_proposals:` block is absent with no explanation.
3. The `capability_context_proof:` block is absent.
4. The worker.reported comment contains a `subagents:` block with named subagents
   when the Plane contract declares an empty `SubAgentRoster`.

**Source:** `FACT(docs/agents/cao.md)` Claude C-Level Worker Pass — checks 3, 4, 1
require `reflection:`, `learning_proposals:`, and `capability_context_proof:` blocks;
`FACT(docs/agents/cao.md)` Enforcement History — `cao.reflection-block-missing` and
`cao.learning-proposals-missing` are active reject codes for dispatcher-v1.2+ runs.

## 4. Hard-Fail Conditions (Exhaustive)

The following nine conditions force overall `FAIL` regardless of how other dimensions
score. They are non-negotiable and cannot be overridden by a high score on other
dimensions.

| # | Condition | Source Dimension | Reject Code |
|---|---|---|---|
| HF-1 | At least one load-bearing factual claim has no citation at all (uncited fact) | D9 | `cmo.unsubstantiated-claim` |
| HF-2 | At least one cited source is outside its class-specific freshness window per the 0.6D table and the claim was not removed | D3 | `cmo.unsubstantiated-claim` (stale variant) |
| HF-3 | The claim-review artifact is absent (claim gate was not run) | D5 | `cao.report-non-pass` |
| HF-4 | The claim-review artifact shows `REGULATED_CLAIM_DETECTED` with `hg3_escalation_required: true` and no Founder approval; the draft was advanced past the gate | D5 | `cao.report-non-pass` + `cmo.unsubstantiated-claim` |
| HF-5 | The writer and editorial reviewer are the same entity (self-review of own draft) | D4 | `cmo.brand-voice-violation` (owner-separation variant) |
| HF-6 | The report artifact contains a public URL to live external content generated by this run, or the producing worker invoked an external publishing API | D6 | `coo.runbook-missing` (publish-without-gate variant); escalate to CEO |
| HF-7 | The producing worker accessed customer-level data (user rows, PII, customer list) | D7 | `coo.runbook-missing` (customer-data-access variant); escalate to Founder (HG-4) |
| HF-8 | The producing worker made a spend change or proposed a spend change through a content-improvement intent card without a [FOUNDER] tag | D7 | `cfo.spend-impact-missing` |
| HF-9 | The `reflection:`, `learning_proposals:`, or `capability_context_proof:` block is absent | D10 | `claude.reflection-missing`, `claude.learning-proposal-missing`, `claude.capability-profile-missing` |

`FACT(docs/agents/cao.md)` — all reject codes listed above are stable identifiers
from the CAO CMO Pass, COO Pass, CFO Pass, and Claude C-Level Worker Pass sections.
The mapping of hard-fail conditions to existing reject codes ensures the controller
verdict schema (Section 5) remains compatible with the canonical CAO verdict shape.

## 5. Controller Verdict Schema (Pasteable)

The controller posts exactly one verdict comment per reviewed artifact. The comment
uses the canonical `controller.verdict` YAML block from `docs/agents/cao.md` Section
"Verdict Comment Shape", extended with the 0.6F rubric-specific fields below.

### 5.1 Full Pasteable Schema

```yaml
controller.verdict:
  version: cao-v0
```

### 5.2 WARN Verdict Usage

A WARN verdict is issued when no dimension scores FAIL and no hard-fail condition is
triggered, but at least one dimension scores WARN. A WARN verdict does NOT advance the
artifact to `ceo:review`. It routes to the owning CMO or CAO planning seat for the
warned dimension to be resolved and resubmitted. CEO/Codex may not release an artifact
with an outstanding WARN verdict without first resolving the WARN and obtaining a
PASS verdict on resubmission.

### 5.3 FAIL Verdict Usage

A FAIL verdict routes the artifact back to the owning `role:cmo` planning seat (or
Founder for HG-4-triggering hard fails). The producing worker must fix the failed
dimensions and resubmit. The controller does not re-defend its FAIL on resubmission;
the resubmission must satisfy the rubric from scratch.

### 6.2 WARN Routing

- **0.6B blog artifact WARN:** → `c-level:cmo:planning` — blog workflow owner resolves
  the warned dimension (e.g., missing persona path in D1, URL-only citation in D2,

- **0.6C social artifact WARN:** → `c-level:cmo:planning` — social queue owner
  resolves and re-enters the affected queue state.
- **0.6D claim artifact WARN:** → `c-level:cao:planning` — CAO seat that produced the

- **0.6E analytics report WARN:** → `c-level:cmo:planning` — CMO resolves the metric
  citation gap or confirms the [FOUNDER] tag is correct before re-submission.

## 9. Reflection

`reflection:`

- The plan-and-report mode mapped cleanly to the scope: this artifact defines a
  deterministic grading harness that a CAO or Codex controller can apply to any 0.6
  Growth wedge packet, without creating content, approving claims, or mutating any
  external system. Every dimension is graded against evidence, not judgment.
- The hardest design discipline was ensuring the rubric grades all five upstream
  artifacts (0.6B–0.6E) without duplicating their rules or creating a sixth approval
  authority. The rubric cross-checks that prior gates ran and passed; it does not re-run
  those gates. This distinction — "verify the gate artifact exists and returned a valid
  verdict" versus "re-adjudicate the gate" — required care in D5 (claim gate compliance)
  and D8 (analytics feedback compliance).
- The WARN tier was added because a binary PASS/FAIL rubric would force a full FAIL
  on minor gaps (e.g., a reviewer-identity field that is blank but separation is
  otherwise demonstrable). WARN gives the controller a signal that the artifact needs
  attention without blocking release for fixable gaps. However, WARN routes back to
  the owning seat before escalating to CEO/Codex — it does not silently advance the
  artifact.
- The nine hard-fail conditions (Section 4) were derived by exhaustively matching the
  acceptance criteria's listed hard-fail triggers to the reject-code matrix in
  `docs/agents/cao.md`. Each hard fail maps to a stable reject code so the
  controller.verdict schema in Section 5 is compatible with the CAO verdict shape
  without introducing new codes.
- The D8 (analytics feedback compliance) dimension required a PASS (n/a) path because
  most claim-review and editorial artifacts do not cite analytics data. The n/a path
  prevents artificial FAILs on artifacts that are out of scope for the analytics loop.
- The routing table in Section 6 was designed to be exhaustive without granting any
  new authority. Every path either returns to a planning seat or escalates to
  CEO/Founder. No path terminates in Plane Done, publish, schedule, spend change, or
  regulated claim approval.

## 10. Learning Proposals

`learning_proposals:` (proposal-only; no durable memory, SOP, skill, workflow or
harness change happens without controller/CEO review)

1. **proposal:** Extract the ten-dimension scoring table shape from Section 3 into a
   generic "Rubric Dimension Template" at `docs/templates/rubric-dimension-template.md`
   so future controller rubrics (department pack controller in 0.8, morning brief
   controller in 0.7) can inherit the PASS/WARN/FAIL three-tier semantics, the
   evidence-per-tier format, and the hard-fail-trigger field without re-defining the
   shape. Risk: premature abstraction before a second rubric is validated in practice.
   Rollback: delete the template; per-rubric dimension tables remain inline.

2. **proposal:** Add a `rubric_version` field to the canonical `controller.verdict`
   YAML shape in `docs/agents/cao.md` so any future rubric evolution (0.6F-v2,
   hypothetical 0.7F controller rubric) is traceable in the Plane audit trail without
   ambiguity. Risk: schema drift if `rubric_version` is interpreted differently across
   CAO sessions. Rollback: remove the field; rubric version tracked in the verdict
   `reason_codes` block only.

3. **proposal:** Cross-reference the 0.6F rubric from the 0.6D Claim/Compliance Gate
   spec (per `docs/releases/0.6d-claim-compliance-gate.md` Learning Proposal #5) and
   from the 0.6E Analytics Feedback Loop spec (Learning Proposal #5), so both specs
   explicitly name the D5 and D8 dimensions as the cross-check hook into those
   respective gates. This would make the cross-reference bidirectional without
   duplicating rules. Risk: coupling three specs to a rubric that may evolve
   independently. Rollback: keep the specs independent; rubric dimensions stand alone.

4. **proposal:** Add a `verdict_WARN` state to the `controller.verdict` YAML in
   `docs/agents/cao.md` alongside the existing `PASS / REJECT / PARK` schema, so the
   three-tier rubric grading (PASS/WARN/FAIL) can be expressed in the canonical
   controller verdict shape without renaming WARN to REJECT. Risk: introducing WARN as
   a new CAO verdict type creates ambiguity with the existing PARK verdict (PARK is a
   deferral, not a quality finding). Rollback: map WARN back to REJECT with a
   `warn_only: true` flag in the `reason_codes` block.

5. **proposal:** Have the 0.6E weekly analytics feedback report include a D-score
   summary for each 0.6 Growth packet that was graded that week (D1–D10 tally, PASS
   rate, WARN count, hard-fail triggers). This would close the measurement loop between
   the editorial controller rubric and the analytics feedback system without requiring
   a new data connector — the weekly report already reads `reports/releases/0.6.0-beta/*.md`
   as M9. Risk: conflating editorial quality grading with operational analytics. Rollback:
   keep the two artifacts separate; the M9 metric reads run outcomes (PASS/REJECT/BLOCKED)
   rather than editorial scores.
