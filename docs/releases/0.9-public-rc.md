# Company.OS 0.9 Public Install Release Candidate

Status: release-candidate notes for `0.9.0-rc.0`
Audience: a remote founder who wants to install Company.OS from the public
GitHub source, follow EVE's guided first-run flow, and update an existing
`0.7.4-rc.0` install.
Last updated: 2026-06-03

## What this is

`0.9.0-rc.0` is the Command EVE remote-install release candidate. It is the
first Company.OS release-candidate line where the sanitized public clone or
public-mirror artifact is the canonical distribution source for a remote
founder. A remote founder can:

- clone the public Company.OS source
- run the public-RC install wrapper against an empty workspace
- generate EVE's first boot packet and an intake from generic signup/report
  seed
- update an existing `0.7.4-rc.0` install through the canonical update path

This is a **guided/self-serve RC, not a stable production product**. It is
not stable unsupported autonomy, not scheduler-default-on, and not an
autonomous publish/schedule/send/spend/release-worker release.

## Install entry point

The supported public install entry point on this RC is the public-RC install
wrapper:

```bash
git clone https://github.com/MathiasHeinke/company-os-public.git
cd company-os-public

node scripts/install/public-rc.mjs install \
  --target /path/to/company-workspace \
  --company "Acme Systems" \
  --website "https://acme.example" \
  --offer "AI operating-system setup" \
  --buyer "founder-led service firms" \
  --founder "Jane Founder" \
  --approval-owner "Jane Founder" \
  --first-department marketing \
  --json
```

This wrapper performs bootstrap dry-run, kit install, intake write, EVE boot
packet generation, update provenance dry-run and a public-RC handoff report
in one bounded flow. Outputs land under the `--target` workspace at:

```text
/path/to/company-workspace/.company-os/onboarding/company-intake.json
/path/to/company-workspace/.company-os/onboarding/eve-boot-packet.json
/path/to/company-workspace/reports/company-os-public-rc/<YYYY-MM-DD>/company-os-public-rc-0.9.0-rc.0.md
```

## Update target

`0.7.4-rc.0` is the supported update target for `0.9.0-rc.0`.

An operator updating an existing `0.7.4-rc.0` install must run the canonical
update command twice in dry-run mode before any non-dry-run apply:

```bash
node scripts/update/company-os-update.mjs check \
  --source /path/to/company-os-public \
  --target /path/to/existing-074-install \
  --write-report --json

node scripts/update/company-os-update.mjs apply \
  --source /path/to/company-os-public \
  --target /path/to/existing-074-install \
  --dry-run
```

Earlier 0.6.x, 0.7.0, 0.7.1 and 0.7.3 installs are not in scope for a direct
one-shot update on this RC. Reach `0.7.4-rc.0` first, then use the 0.7.4 →
0.9.x update path.

- 0.7.4 Supergoal Factory substrate: Execution Ladder, State-Truth Pass,
  HG-2.5 Merge Readiness Pack, Post-Worker Quality Registry, lower-worker
  audit/hotfix capability profiles.
- 0.7.3 Command EVE v1.3 governance spine: Supergoal Execution Ladder,

  Connector Integration Harness.
- 0.7.1 self-serve install/onboarding/update smoke baseline.
- 0.7.0 EVE / AionUI / Hermes operator-shell packaging substrate.
- 0.6.5 Plane-first runtime substrate: capability registry, sandbox workspace
  guard, Raindrop managed-call coverage.
- Public-RC install wrapper (`scripts/install/public-rc.mjs`) for remote
  founders.
- Public-mirror coverage for `registries/domain-packs/**`,
  `registries/plane-templates/**` and `registries/quality/**` while private
  capability and inference registries remain stripped or replaced with
  examples.

## What is explicitly NOT in this RC

`0.9.0-rc.0` is a release candidate, not stable production. The following
remain gated past this cut:

- GitHub tag creation, GitHub release publishes and merges to public main.
- Scheduler-default-on autonomy and unattended self-serve installs.
- Autonomous publish, schedule, send, outreach, spend or production-write
  actions.
- Regulated medical, Rx, legal, financial, security or privacy claim
  approvals.
- Plane `Done` transitions by workers, CAO or controller.
- Direct controller spawning of follow-up workers.
- Production deploys and schema/RLS/auth changes.
- Copying the private source tree (`reports/`, `metrics/`, internal evidence)
  into the public artifact.

## Release gate

The minimum release gate for promoting `0.9.0-rc.0` to a public-published
release (a separate HG-2.5 / HG-3 action, not part of this baseline) is
documented in `docs/releases/versioning.md`. The release-baseline scope here
is limited to version identity, changelog framing, README/release-doc framing
and remote-install entry points. Public publishing, tagging and GitHub
release uploads remain CEO/Founder HumanGate decisions.

## Boundary statement

`0.9.0-rc.0` is the Command EVE remote-install release candidate. It is a
guided/self-serve RC for a remote founder to try, not a stable production
release. Promotion to a stable line requires CEO release evidence, repeated
non-founder pilot evidence and explicit HumanGate sign-off.
