# Command EVE Offer v1

Status: public product strategy substrate
Use for: positioning the first paid Command EVE offer without claiming stable
self-serve autonomy before the release gates prove it
Last updated: 2026-06-04

## Decision

Command EVE sells a managed operator outcome first and turns repeated delivery
evidence into self-serve product later.

The first cash motion is not "download this and everything runs itself." The
first sellable motion is:

```text
AI/Company Operating Audit
-> first department wedge
-> guided Command EVE setup
-> recurring gated operating loop
-> reusable templates, skills and worker contracts
-> later self-serve release gate
```

This protects trust. The product may install from GitHub, but the commercial
promise stays honest: guided alpha or release candidate until install,
support, security, privacy, update and autonomy gates are proven outside the
core team.

## Positioning

Command EVE is the operator shell for Company.OS: a founder-facing AI chief of
staff that helps understand the company, turns intent into CEO/C-Level/worker
contracts, and keeps the operating loop visible through evidence, gates and
reports.

Recommended public frame:

```text
Command EVE helps you install an AI operating system for your company, starting
with one high-value department lane and explicit human gates.
```

Avoid overclaiming:

- no stable unattended self-serve claim before the stable baseline gate;
- no autonomous publishing, outreach, spend, production write or Plane Done
  claim;
- no claim that EVE replaces founder judgment;
- no claim that a generic install knows the company before onboarding or
  connector evidence exists.

## Initial ICP

Start with narrow operator-friendly customers:

- solo or boutique consultants;
- fractional experts;
- small agencies;
- founder-led expert businesses;
- teams that already feel the coordination cost of content, sales, operations
  or client delivery.

The first buyer should have a clear first lane. Good first lanes:

- content and LinkedIn operations;
- offer and proposal operations;
- client update/reporting operations;
- internal knowledge and operating wiki;
- lead signal review and human-gated outreach drafts.

Regulated, HR, finance, legal, medical, safety-critical or high-volume
customer-action lanes are later gates, not first install promises.

## Offer Ladder

| Offer | Role | Boundary |
|---|---|---|
| AI/Company Operating Audit | paid diagnosis and lane selection | produces score, risks, first wedge and setup plan; no automation promise |
| Founder OS | first guided lane | one department wedge, templates, onboarding, manual gates |
| Growth OS | managed operating loop | recurring CMO/COO/CEO lane, reports, approval queue, update rhythm |
| Managed AI Ops | premium managed operation | multiple lanes only after security/privacy/support gates |

Pricing is intentionally implementation-dependent. Public docs may include
example tiers, but customer-specific pricing and founder-discretion discounts
do not belong in the reusable release baseline.

## Self-Serve Flip Gate

Self-serve may be called stable only when at least these conditions are true:

1. a fresh public install succeeds without private-path knowledge;
2. update from the previous public version is proven;
3. the onboarding flow can ingest company seed facts, show what it knows, ask
   only for missing context, and persist an inspectable boot packet;
4. support, privacy, data retention, connector auth and kill-switch boundaries
   are documented;
5. at least one non-founder operator can run the install and first lane without
   private chat-copy-paste;
6. release tags/uploads and public distribution checks are HG-2.5 approved.

Until then the correct label is guided alpha or release candidate.

## Proof Loop

Marketing should demonstrate Command EVE by using Command EVE:

- source inventory;
- founder voice and belief check;
- content draft;
- claim/safety gate;
- human approval;
- publication queue;
- performance feedback;
- lessons back into templates.

The visible claim is not "fully autonomous." The visible claim is "the system
can prepare and operate the lane up to explicit approval gates with evidence."

## Public Boundary

Keep in public:

- reusable offer structure;
- score/questionnaire pattern;
- lane templates;
- worker contracts;
- install and update flow;
- release gates.

Keep out of public:

- private customer lists;
- founder-specific revenue assumptions;
- raw audience exports;
- private sales notes;
- customer credentials or connector state;
- product-specific health, finance, legal or regulated claims.
